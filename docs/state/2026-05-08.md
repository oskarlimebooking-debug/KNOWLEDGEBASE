### Phase D — Personality system (D1)

- `PERSONALITIES` const with all **12 personalities** (7 Headway core +
  5 ThesisCraft additions): `professor, researcher, hype, contrarian,
  critic, unhinged, nurturing, storyteller, meme, practitioner,
  philosopher, journalist`.
- Each row carries `code`, `label`, `avatar`, `voice` (paragraph
  description used inside the prompt and writeup), 4-handle `usernames`
  pool, `imageStyle` hint, and an `origin` flag.
- `DEFAULT_BOOK_PERSONALITIES` rotation (the 7 Headway core) is what
  ships for `book` source kind today. The other 4 source kinds will
  enable their alternate rotations in Phase I when source generalization
  lands.
- `personalityByCode(code)` resolves a code → record (defaults to
  professor).

### Phase D — Feed prompt + JSON parser (D2 / D3)

- `FEED_PROMPT` template (~2.5k chars) lives next to the personality
  table. Variables: `{{title}}, {{content}}, {{postCount}},
  {{personalityCount}}, {{personalityList}}`. Cadence, structural
  requirements (3 image, 5–6 link, 3–4 viral, 3–4 thread), output
  schema, and username constraints all in one block.
- Setting key `prompt_feed` lets users override.
- `parseFeedJson(raw, apiKey)` runs three tiers:
  1. Reuse `parseSafeJson` (handles fences + walks first-object braces).
  2. Splice the `"posts": [ ... ]` array out using a brace-aware walker
     when the wrapping object is malformed.
  3. AI-assisted repair via `attemptFixChapterFeedJson(rawBroken,
     apiKey)` — sends the broken text back to Gemini in JSON-mode with
     "preserve as much content as possible; drop irreparable posts".
- Throws with a usable message if all three fail.

### Phase D — Feed generator (D2 / D8)

- `generateFeed(chapter, opts)` reads `feedPostCount` (clamped 10–40),
  builds the prompt, calls Gemini with `temperature: 0.85,
  maxOutputTokens: 32768, timeoutMs: 180000`, parses with
  `parseFeedJson`, normalizes each post via `normalizeFeedPost`.
- Cached as `feed_<chapterId>` in the `generated` store via the
  standard `loadGenerated('feed', ...)` flow.
- `generateMorePosts(chapter, existingFeed)` — appends ~half-count more
  posts with a "do NOT repeat these openings" instruction listing the
  first 12 existing post topics. Re-uses the same prompt template.
- `normalizeFeedPost` enforces shape: stable id `p<n>`, personality →
  resolved record, username clamped to the personality's pool, hashtags
  capped at 5, fake stats baked in via `randInt`,
  `getRandomTimeAgo()` for human-readable timestamps.
- `formatNumber(n)` → `1.2k` / `45.3k` / `2.1M`.

### Phase D — Feed renderer (D4)

- `renderFeedMode(host, ctx)` — needs-key state, loading, error, then
  hands off to `renderFeedView`.
- `renderFeedView` paints a toolbar (count + top-4 personality
  histogram) + a `.feed-list` of `.feed-card`s, then wires each
  card's actions in `wireFeedCards`.
- `renderFeedCardHtml(post, isLiked)` produces a Twitter/X-shaped card:
  - `.feed-avatar` 40×40 emoji circle, `.feed-handle` username +
    @handle + time.
  - Viral and Thread pill badges, color-coded.
  - Content split on newlines with `formatPostLine` highlighting
    `#hashtag` matches.
  - Optional image block (rendered img if `imageUrl` present, else
    "🖼 Generate image" placeholder).
  - Optional `📖 Read deep-dive: <topic>` button when `hasLink`.
  - Action row: 💬 reply / 🔁 retweet / 🤍-❤ like / 🔗 share / 👁 views.
- Reply / retweet / share are visual-only and surface a "Visual only —
  this is your fictional feed 🤖" toast (the spec calls this out as
  optional polish; better than silent buttons).

### Phase D — Image generation (D5)

- `callGeminiImage(prompt, apiKey)` tries three image-capable Gemini
  models in fallback order (`gemini-2.5-flash-image` →
  `gemini-2.0-flash-preview-image-generation` →
  `gemini-2.0-flash-exp-image-generation`). Sends
  `responseModalities: ['TEXT', 'IMAGE']` and pulls the first
  `inlineData.data` part out of the response. Returns a
  `data:image/...;base64,...` URL.
- `generateFeedImageForPost(post)` injects the personality's
  `imageStyle` as a prefix and "Vertical 4:5 aspect ratio. No text
  overlays." as a suffix to keep results consistent.
- On success the URL is mutated onto `post.imageUrl`, the cached row is
  re-saved, and the card is re-rendered. On failure the placeholder
  comes back with a friendly toast.

### Phase D — Deep-dive writeup modal (D6)

- New top-level modal `#modal-writeup` (`.writeup-modal`, 760 px max,
  86vh tall on desktop, fullscreen on mobile).
- `WRITEUP_PROMPT` injects the personality voice paragraph + topic so
  Gemini holds character for an 800–1500 word markdown article with
  3–5 subheadings.
- `openWriteupModal(post, ctx)` sets the head (avatar + label + topic),
  shows a loader, hits the cache (`writeup_<chapterId>_<postId>_<personality>`),
  generates if absent, renders via `mdRender` inside `.markdown.writeup-article`.
- Setting key `prompt_writeup` lets users override.

### Phase D — Like / retweet state (D7)

## Locked decisions

- **Personality codes are final.** Adding new ones later is fine
  (Phase Q's user-defined personalities); renaming would break
  cached feeds.
- **Image storage is inline base64** for now (data URL on
  `post.imageUrl`). The size cost is acceptable when only 3 of 20
  posts have images; Phase M will move this to OPFS file references
  if/when audio-and-image storage starts to matter.
- **`feed_<chapterId>` cache shape** locked at `{ posts: [...],
  generatedAt, chapterId }` — Phase L's book-feed and multi-book-feed
  will reuse `posts[]` items verbatim and add their own wrapper.
- **Reply / retweet / share are visual-only.** Like / image / writeup
  are persisted; the others surface a friendly toast.
