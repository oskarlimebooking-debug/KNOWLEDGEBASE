### Phase B closed-out gaps

#### Pattern-based chapter detection (B3)

- `detectChapterPatterns(text)` runs four regex tiers in priority order
  on the extracted text **before** the word-count splitter:
  1. `Chapter <num|roman|word>` (multilingual-friendly digit/roman/written-word).
  2. `Part <num|roman|word>`.
  3. `Section <num|roman>`.
  4. Bare numeric / Roman headings on their own line (`1.`, `I.`, `II.`).
- First pattern with **≥ 2 matches** wins; chapter slices are cut at
  match boundaries; slices < 200 chars are filtered out (drops TOC false
  positives).
- The matched heading line becomes the chapter title (capped at 80
  chars). The heading text is stripped from the slice body so the Read
  view doesn't echo "Chapter 3" inside the prose.
- If no pattern fires (or fewer than 2 matches survive filtering), the
  word-count splitter is used as before.
- `book.detectionMethod` ∈ `{ chapter, part, section, numeric, wordcount }`
  is recorded so future debugging / re-splits know which path produced
  the layout. Smoke-tested live with synthetic fixtures — all four tiers
  fire, plain prose returns null.

#### AI-enhanced chapter titles

- After splitting (pattern or word-count), `enhanceChapterTitlesWithAI`
  fires **once** with a numbered list of 80-word openings and current
  titles. Single batch Gemini call, JSON-mode, temperature 0.4.
- Returned titles are validated (length 3–80) and only adopted when the
  array length matches the chapter count. Any failure is swallowed
  silently — the pattern/derived title is the fallback.
- Trigger: `hasAnyAIProvider()` is true.
- Skipped silently when no key is configured (additive behavior — never
  blocks book import).

#### Format-all-chapters (B10 completion)

- `runFormatAllChaptersDialog()` triggered from a new ✨ icon button on
  the **book-detail header**.
- Filters chapters to those without `formattedHtml`, confirms with the
  user (one Gemini call per chapter), runs sequentially.
- Bottom progress banner shows "Formatting N / M: <chapter title>" with
  a Cancel button. Cancel stops cleanly between chapters.
- Reports done / failed counts on completion. Reopens the book detail
  view after running so the chapter list refreshes.

### Phase C — Mind Map (C1)

- `generateMindmap(chapter)` — strict JSON `{ center, branches[] }`,
  3–5 branches, 1–3 sub-branches each, 0–3 items per sub. Defaults
  applied to enforce shape (slice/length caps).
- `renderMindmapSvg(data)` — hand-rolled SVG, viewBox 900×640, center
  rounded-rect node, branches radiating at even angles around it,
  connectors color-coded from the 5-color palette
  (`#FF6B6B #4ECDC4 #45B7D1 #FFA07A #98D8C8`), sub-branches stacked
  vertically off each branch with dashed connectors, items as text
  beneath each sub-box.
- `escapeXml` helper isolates SVG text injection (separate from
  `escapeHtml` because `'` and `"` matter in attributes too).
- Plugged into the standard `renderAIMode` cache pattern — cached as
  `mindmap_<chapterId>` and survives reload. Live-verified with a
  planted fake row: 7 rects + 6 lines + correct center text.
- Custom mindmap prompt setting key reserved (`prompt_mindmap`) but
  not yet exposed in the Settings textarea grid.

### Phase C — Conversation manager (C2)

`mountConversation(host, opts)` — single reusable component for any
chat-style mode.

- State: `messages: Array<{ role, content, ts }>` where `role ∈ {user,
  ai, system, error}`.
- DOM: `.chat-messages` (flex-column scroll) with bubble layout
  (`.chat-msg.user` right, `.chat-msg.ai` left, `.chat-msg.system`
  centered + dashed border, `.chat-msg.error` red-tinted).
- Input: textarea auto-resizes up to 160px; **Enter sends, Shift+Enter
  newlines** (per spec).
- Loading: three-dot bouncing typing indicator while waiting for AI.
- `formatChatMessage(text)` — inline markdown only (`**bold**`,
  `*italic*`, `` `code` ``, line breaks). HTML-escaped first — no XSS
  risk through chat output.
- **Simulated streaming** (C6 plumbing): `opts.simulateStream=true`
  chunks the final response into 50-char slices with a 20 ms delay so
  the UX feels alive even though Gemini's REST endpoint isn't true SSE.
  When Phase H wires Merlin's SSE, this exact code path drops in.
- `opts.systemPrompt` is the preamble; `opts.contextProvider()` returns
  reference content (chapter / book) injected fresh each turn.
- `opts.onHistoryChange(msgs)` fires after each AI turn for callers
  that want to persist (Socratic does; Ask does not).
- Returns `{ reset(), getMessages() }`.

### Phase C — Socratic mode (C3)

- System prompt enforces "never give answers, probe with questions,
  build on previous answers, synthesize only if stuck after 5 turns,
  never break character".
- Persisted in `generated` store as `socraticHistory_<chapterId>`. On
  re-open, history is restored. Auto-saves after each AI turn via
  `onHistoryChange`.
- Reset button confirms then wipes the row.
- Chapter content truncated to 12 000 chars (with `[…truncated…]`
  marker) to keep the system prompt + history + chapter under typical
  Gemini context budgets.

### Phase C — Chapter Ask mode (C4)

- System prompt: "Answer ONLY from the chapter; if not in the chapter
  say so explicitly."
- Conversation **not** persisted — each open starts fresh per spec
  (search-style interaction). `onHistoryChange` omitted.
- Chapter truncated to 18 000 chars (more than Socratic since no
  history takes up budget).

### Phase C — Book Ask modal (C5)

- 💬 icon button on book-detail header opens a full-screen overlay
  (max-width 720px desktop, 100vh mobile) titled `Ask "<book>"`.
- "New conversation" button calls `conv.reset()`; "Close" hides the
  modal but keeps history alive in the JS state until close-then-reopen
  (intentional — matches search-style interaction).
- `buildBookContext(book, chapters, 22000)`:
  - **Small books** (combined chapters ≤ 22 000 chars): full text with
    `## <title>` per chapter.
  - **Large books**: chapter list + per-chapter blurbs preferring cached
    Phase B summaries when available, else 600-char openers. Falls back
    gracefully when no summaries are cached.
