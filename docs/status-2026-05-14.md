## Feature details

### N2 — In-text annotations

**IDB store: `annotations`**
```
{
  id: 'ann_<ts>_<rand>',
  chapterId, bookId,
  start: number,   // char offset from article textContent start
  end: number,
  color: 'yellow' | 'pink' | 'green' | 'blue',
  text: string,    // selected text (max 300 chars)
  note: string,    // optional note
  createdAt: ISO
}
```

**Highlight flow:**
1. User selects text in Read view → `mouseup`/`touchend` fires on `#mode-host`
2. `_showAnnMenu()` positions floating menu near selection with 4 colour swatches
3. Clicking a swatch (or Save with a note) calls `_createAnnotation(color, note)`
4. Annotation saved to IDB; existing marks stripped from DOM; `_applyAnnotationsToDOM`
   re-applies all annotations in reverse-offset order (to preserve positions)

**Persisting across renders:**
- Offsets are character positions within the article's `textContent`
- `_offsetToRange` walks all text nodes via `TreeWalker` to reconstruct the Range
- Marks are applied in descending `start` order so earlier offsets stay valid

**Sidebar:**
- 🖊 button in chapter header opens/closes a 300 px slide-in panel
- Lists all annotations sorted by position with coloured left border
- Clicking a chip scrolls to the mark (smooth) and flashes the accent outline

**Mark popup:**
- Clicking a `<mark>` shows an inline popup with the selected text, any note, and a Delete button
- Delete removes from IDB and replaces mark with plain text node

## Known gaps / follow-ups

1. **Cross-paragraph selections** — `surroundContents` may fail if selection crosses `<p>` boundaries; the `extractContents` fallback creates a `<mark>` wrapping block-level elements (browsers handle it but it's non-standard). A multi-segment approach would be cleaner.
2. **Annotation offsets vs. formattedHtml** — offsets are computed from the rendered article's textContent. If a chapter is AI-formatted after annotations are created, offsets may drift. Best practice: annotate after formatting.
3. **Heatmap dark-palette** — cells still hardcoded to dark-theme greens (noted from previous session).
4. **Mobile selection menu** — the menu positions at `range.getBoundingClientRect()` which may land off-screen on small phones; clamped to `max(8, x)` and `min(x, vw-220)` horizontally but vertical overflow is not handled.
